load 与 initialize方法小记

发表于 2016-05-28

iOS程序启动后会加载所有的类

程序经过编译，链接之后以二进制的形式存在，一旦点击运行，所有的类对象会加载到内存中。

load与initialize到底是属于谁的方法

它们都是NSObject的类方法。

load方法的调用时机

load方法是在类加载后被调用，因此它比系统的main方法还早。load方法仅仅会调用一次。
load方法是系统来调用，而不应该是自己显式调用。

initialize方法的调用时机

在runtime向类发送第一个方法前，会调用一下initialize方法。如果始终不调用类的任何方法，initialize方法也不会执行。

load，initialize方法都只会调用一次

load调用且只调用一次。
initialize在第一个类的方法前也只会调用一次。如果从来不使用这个类。不会调用initialize方法

load，initialize方法的使用场景

load是程序加载到内存后就马上调用的，它适合做方法交换
initialize是在类第一个方法被使用的时候调用，它适合做全局配置。如统一导航栏的颜色。

superview subview category

load方法调用的顺序：父类(Superclass)的方法优先于子类(Subclass)的方法，类中的方法优先于类别(Category)中的方法

NSTextAttachment实现图文混排

发表于 2016-05-26

需求场景

设计师希望在UILabel后面添加一个关于的图标。引导用户去点击这个Label.由于Label的文案接口返回，无法得知Label文案结束的位置，因此无法决定图标摆放在哪里。

NSAttributeString与NSTextAttachment

NSTextAttachment，顾名思义，文本附件。
UILabel有一个attributeString属性，我们可以称为带属性的字符串。通过设置该属性值，可以实现文案不同颜色显示，文案带下划线或者带删除线等。在iOS7中，NSAttributeString甚至可以实现简单的图文混排，关键的核心是将文本附件转换为NSAttributeString.和一般的NSAttributeString拼接在一起。那么我们布局按钮的事情直接转换为布局一个简单的字符串了。说了那么多废话，让我们直接来代码演示一下。下面展示了不同长度的字符串下图文混排。

@interface ViewController ()

@property (weak, nonatomic) IBOutlet UILabel *label;
@property (weak, nonatomic) IBOutlet UILabel *label2;
@end

@implementation ViewController

- (void)viewDidLoad {
    [super viewDidLoad];
   // 在变化的标签后添加图案
    NSString *text = @"我想实现简单的图文混排";
    NSMutableAttributedString *attributeString = [[NSMutableAttributedString alloc] initWithString:text];

    NSTextAttachment *attachment = [[NSTextAttachment alloc] initWithData:nil ofType:nil];
    attachment.image = [UIImage imageNamed:@"about"];
    attachment.bounds = CGRectMake(10, -1, attachment.image.size.width, attachment.image.size.height);

    NSAttributedString *string = [NSAttributedString attributedStringWithAttachment:attachment];
    [attributeString insertAttributedString:string atIndex:text.length];
    self.label.attributedText = attributeString;


    // 在变化的标签后添加图案
    NSString *text2 = @"我想实现简单的图文混排我想实现简单的图文混排";
    NSMutableAttributedString *attributeString2 = [[NSMutableAttributedString alloc] initWithString:text2];

    NSTextAttachment *attachment2 = [[NSTextAttachment alloc] initWithData:nil ofType:nil];
    attachment2.image = [UIImage imageNamed:@"about"];
    attachment2.bounds = CGRectMake(10, -1, attachment2.image.size.width, attachment2.image.size.height);
    NSAttributedString *string2 = [NSAttributedString attributedStringWithAttachment:attachment2];
    [attributeString2 insertAttributedString:string2 atIndex:text2.length];
    self.label2.attributedText = attributeString2;

}

RunLoop之作用篇

发表于 2016-05-25

命令式执行与事件驱动

首先我们来看看命令行代码 

int main(int argc, char *argv[]) {
    NSLog(@"hello world");
    return 0;
}

执行完了就程序结束。我们回想一下平时使用的手机应用程序，并非如此。可以不断的响应用户的操作，用户不做任何操作的时候，程序也不会终止。为了实现上面两个目标，实际的手机应用程序用伪代码来表示基本可以认为是这样的

int main(int argc, char *argv[]) {
    while(AppIsRunning) {
        id whoWakeMe = SleepForWakingup();
        id event = GetEvent(whoWakeMe);
        Handle(event);
    }
    return 0;
}

通过死循环。如果没事的时候，就处于休眠状态。如果有事件，就将其唤醒，然后去处理这个事件。

为什么需要RunLoop

  1. 让程序一直运行并接受用户输入
  2. 决定何时处理哪些事件Event
  3. 调用解耦？
  4. 节省CPU时间，没事的时候闲着，不让cpu执行它。

RunLoop到底是什么？

RunLoop 实际上就是一个对象，这个对象管理了其需要处理的事件和消息，并提供了一个入口函数来执行上面 Event Loop 的逻辑。线程执行了这个函数后，就会一直处于这个函数内部 “接受消息->等待->处理” 的循环中，直到这个循环结束（比如传入 quit 的消息），函数返回。

OSX/iOS 系统中，提供了两个这样的对象：NSRunLoop 和 CFRunLoopRef。
CFRunLoopRef 是在 CoreFoundation 框架内的，它提供了纯 C 函数的 API，所有这些 API 都是线程安全的。
NSRunLoop 是基于 CFRunLoopRef 的封装，属于Foundation框架。提供了面向对象的 API，但是这些 API 不是线程安全的。

RunLoop 与线程的关系

首先，iOS 开发中能遇到两个线程对象: pthread_t 和 NSThread。过去苹果有份文档标明了 NSThread 只是 pthread_t 的封装，但那份文档已经失效了，现在它们也有可能都是直接包装自最底层的 mach thread。苹果并没有提供这两个对象相互转换的接口，但不管怎么样，可以肯定的是 pthread_t 和 NSThread 是一一对应的。比如，你可以通过 pthread_main_np() 或 [NSThread mainThread] 来获取主线程；也可以通过 pthread_self() 或 [NSThread currentThread] 来获取当前线程。CFRunLoop 是基于 pthread 来管理的。

线程和 RunLoop 之间是一一对应的，其关系是保存在一个全局的 Dictionary 里。线程刚创建时并没有 RunLoop，如果你不主动获取，那它一直都不会有。RunLoop 的创建是发生在第一次获取时，RunLoop 的销毁是发生在线程结束时。你只能在一个线程的内部获取其 RunLoop（主线程除外）。

Basic

实现三个视图redView,greenView,blueView.其中red,green占一行。等高等宽。blueView占下一行。高度以red，green也相等。red,green,blue和父视图之间的间隔还有他们两两之间间隔都是10.

- (instancetype)init {
    self = [super init];
    if (self) {
        UIView *greenView = UIView.new;
        greenView.backgroundColor = UIColor.greenColor;
        greenView.layer.borderColor = UIColor.blackColor.CGColor;
        greenView.layer.borderWidth = 2;
        [self addSubview:greenView];
        self.greenView = greenView;

        UIView *redView = UIView.new;
        redView.backgroundColor = UIColor.redColor;
        redView.layer.borderColor = UIColor.blackColor.CGColor;
        redView.layer.borderWidth = 2;
        [self addSubview:redView];
        self.redView = redView;

        UIView *blueView = UIView.new;
        blueView.backgroundColor = UIColor.blueColor;
        blueView.layer.borderColor = UIColor.blackColor.CGColor;
        blueView.layer.borderWidth = 2;
        [self addSubview:blueView];
        self.blueView = blueView;

        UIView *superview = self;

        [greenView mas_makeConstraints:^(MASConstraintMaker *make) {
            make.left.equalTo(superview).offset(10);
            make.top.equalTo(superview).offset(10);
            make.width.equalTo(redView);
            make.height.equalTo(@[redView, blueView]);
        }];

        [redView mas_makeConstraints:^(MASConstraintMaker *make) {
            make.left.equalTo(greenView.right).offset(10);
            make.top.equalTo(greenView);
            make.width.equalTo(greenView);
            make.right.equalTo(superview).offset(-10);
        }];

        [blueView mas_makeConstraints:^(MASConstraintMaker *make) {
            make.top.equalTo(redView.bottom).offset(10);
            make.left.equalTo(superview).offset(10);
            make.right.equalTo(superview).offset(-10);
            make.bottom.equalTo(superview).offset(-10);
        }];
//        [self setNeedsUpdateConstraints];
    }
    return self;
}

masonry可以结合xib一起使用

如果我在xib中已经布局了UI，而且建立了autolayout，当程序运行中需要动态修改autolayout，可以直接使用autolayout的API来修改和删除，更极端的，也可以直接使用masonry来删除

masonry,UITableView与autolayout

iOS8以上，对于autolayout，不可以根据子视图的约束动态计算tableViewCell的高度.(注意一般情况下的autolayout的子视图对父视图的约束不会影响父视图，只能影响子视图).tableViewCell通过一定的设置可以开启这个功能。
而iOS8以下的系统。autolayout就无能为例了。
我们先给出iOS8下的解决方案。这种子视图改变，动态改变自身tableViewCell高度的苹果叫做 Self-Sizing Table View Cell。

tableView.estimatedRowHeight = 85.0
tableView.rowHeight = UITableViewAutomaticDimension

首先第一句话，给一个预估高度，等会儿自动布局的会在这个值的基础上再进行调整。第二句话，是开启autolayout的自动布局功能。甚至连高度回调都不需要，完成了整个布局控件的实现。

如果要适配iOS7，那么这种方法就不奏效了。我们依然可以对子视图使用autolayout来约束子视图和父视图之间位置。而对于tableViewCell本身。可以用frame算出高度后再返回。比较常规的实现是写一个类方法。

Masonry手写约束原理篇

发表于 2016-05-24

1.Masonry是什么

  • Masonry是一个轻量级的布局框架
  • Masonry是对AutoLayout的封装，用链式语法实现了比NSLayoutConstraints更准确和可读性更好的接口
  • Masonry 同时支持 iOS 和 Mac OS X两个平台

2.AutoLayout公式与Masonry的等价形式

在真正学习Masonry前，我们先退后一步 想一想，Masonry是AutoLayout的封装，那么为什么iOS要引入AutoLayout了，即AutoLayout想解决什么问题？

  • 更容易适配不同分辨率设备的屏幕(iPhone 6 Plus, iPhone 6, iPhone 5s/5, iPhone 4s/4)
  • 当设备旋转时不需要做额外处理
  • 使用constraint来描述布局逻辑，更利于理解和清晰

我们就要追本述源，看看AutoLayout是怎么样布局的。实际上AutoLayout对布局仅仅记住一个公式就很明了了。

item1.attribute1 = item2.attribute2 * mutilplier + constant;

上图中为了布局BlueView和RedView用了一个简单的公式，推广来说，在autolayout中。item1和item2总是分别表示视图1和视图2.。attribute1,attribute2表示视图属性。这些视图属性可以是表示视图的top,left,bottom,right.也可以表示尺寸的width,height.表示中心点的CenterX,CenterY等等，我们没有必要一一记住他们。当使用的时候查询就好了。mutilplier是系数。这个常常使用在需要等分子视图的场景。constant可以认为是偏移量。=表示关系,这里的关系是equal,实际上还有>=, <=。在Masonry中，我们可以一个简单的例子来看看它是如何体现AutoLayout公式的。

UIEdgeInsets padding = UIEdgeInsetsMake(10, 10, 10, 10);

[view1 mas_makeConstraints:^(MASConstraintMaker *make) {
    make.top.equalTo(superview.mas_top).with.offset(padding.top);         make.left.equalTo(superview.mas_left).with.offset(padding.left);
    make.bottom.equalTo(superview.mas_bottom).with.offset(-padding.bottom);
    make.right.equalTo(superview.mas_right).with.offset(-padding.right);
}];

这个Demo中，make.top.equalTo(superview.mas_top).with.offset(padding.top); 其实是

view1.top = superview.top * 1 + padding.top（即constant);

3.Masonry常用的方法和属性

当我们使用Masonry的时候，几乎总是以这三个方法打交道。它们都是UIView的category方法，这种category的实现方式对于第三方库十分常见。如SDWebImageView中也会对UImageView UIButton写分类方法。它们的声明和实现都在
View+MASAdditions.h中.即下面这三个方法

// 添加约束
- (NSArray *)makeConstraints:(void(^)(MASConstraintMaker *make))block;
// 更新约束
- (NSArray *)updateConstraints:(void(^)(MASConstraintMaker *make))block;
// 重设约束，清空约束后再添加
- (NSArray *)remakeConstraints:(void(^)(MASConstraintMaker *make))block;

那么这三个方法是使用场景又是什么了？
谈到场景的时候，首先要明确几点。什么是不合理的约束

  • Unsatisfiable Layouts. Your layout has no valid solution。
    可能是缺少约束。导致iOS无法判断UIView到底该显示在哪里。或者两种或者多个约束冲突。
  • Ambiguous Layouts. Your layout has two or more possible solutions
    两种或者多种有效的解决方案。
  • Logical Errors. There is a bug in your layout logic
    约束建立成功，但是解决方案不是你想要的结果。

方法确定了到底使用哪个约束，实际上每个方法都接受一个block变量。block变量的定义中操作约束的attribute来完成约束，Masonry支持下面的一些属性

@property (nonatomic, strong, readonly) MASConstraint *left;
@property (nonatomic, strong, readonly) MASConstraint *top;
@property (nonatomic, strong, readonly) MASConstraint *right;
@property (nonatomic, strong, readonly) MASConstraint *bottom;
@property (nonatomic, strong, readonly) MASConstraint *leading;
@property (nonatomic, strong, readonly) MASConstraint *trailing
@property (nonatomic, strong, readonly) MASConstraint *width;
@property (nonatomic, strong, readonly) MASConstraint *height;
@property (nonatomic, strong, readonly) MASConstraint *centerX;
@property (nonatomic, strong, readonly) MASConstraint *centerY;
@property (nonatomic, strong, readonly) MASConstraint *baseline;

我们建立的约束一旦建立后不再有变化，且并不会出现Unsatisfiable Layouts，Ambiguous Layouts，Logical Errors。仅仅使用makeConstraints就好了。

当我们使用makeConstraints建立的约束是有效的，用户的某个事件需要更改UIView的位置，此时需要做到两步。

1.调用 [self setNeedsUpdateConstraints];
2.重写 updateConstraints方法（结尾添加[super updateConstraints]）,并且在这个方法的实现中调用updateConstraints或者remakeConstraints。前者是保持原来约束情况下，更新约束的关系。后者是清空约束后重新添加约束.还要注意的是remakeConstraints的调用并不意味着你必须先调用updateConstraints方法。实际上,remakeConstraints也可以单独调用。

4.Masonry链式语法

链式语法的核心是A.fun1.fun2.fun3…funN.每个fun的输出返回的对象依然是A本身。
在Masonry中,MASConstraintMaker类充当一个工厂的角色。以make.top.equalTo(superview.mas_top).with.offset(padding.top);为例，make.top返回的是MASConstraint对象。此后的equalTo(superview.mas_top)，依然返回MASConstraint对象。所有的.fun都返回MASConstraint对象。显然这样的写法非常紧凑，易读。
