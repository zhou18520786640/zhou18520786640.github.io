UICollectionView Basic

发表于 2016-03-20

UICollectionView 作用

UICollectionView是增强版的UITableView,可以认为是方便实现一行多个cell的tableView。另外利用自定义布局的配置，可以实现更加灵活的布局展示。

UICollectionView的组成

UICollectionView其实不单单指的UICollectionView的类。而是一系列类和协议的总称。下面我们分标题一一介绍。

cells

类似UITableViewCell,不过在collectionview中一行可以显示多个collectionviewcell,要定制化它，最佳实践也和UITableViewCell一样，子类化collectionviewcell.这个cell的是数据驱动的，即要展示什么内容并不是由自己负责，而是提供数据的类。要创建多少cell也是根据数据决定，数据item的个数决定了view的个数。在UITableView中，这个类就是遵从UITableViewDataSource的类。在UICollectionView中这个类就是遵从UICollectionViewDataSource的类。对于UICollectionViewDataSource 协议，我会在UICollectionViewDataSource中对其功能做详细的介绍。另外和tableview一样。对于用户操作cell产生的一些行为，我们也移交给遵从另外一个协议的类。也许你也猜到了。这个类就是UICollectionViewDelegate.实际来说，学习UICollectionView最基本的元素就是UICollectionViewCell。而cell的数据来源于UICollectionViewDataSource,其UICollectionViewDataSource协议的实现才是我们的关注重点。如果用户还需要操作cell后的一些反馈，那么就依靠UICollectionViewDelegate来了。从设计思想来说。我们把内容逻辑的处理，展示内容，cell的反馈处理。分别由三个类来承担了。达到了去耦的目的。这对于比较复杂视图是需要的。而普通的UILabel,UIButton就不需要这样复杂的设计了。

supplementary views

和collectionviewcell类似，supplementary views是数据驱动的。和collectionviewcell，不同这个对象并不是必须的。collectionviewcell的创建多少个和要显示的数据内容也依然依靠的是遵从UICollectionViewDataSource的类。在UICollectionViewDataSource协议中，
- collectionView:viewForSupplementaryElementOfKind:atIndexPath:是个可选方法。其实通俗的讲。supplementary views就是tableview的header和footer.一个section可以配置一个section或者footer。至于标示header或者footer就依靠的是viewForSupplementaryElementOfKind的参数值

Decoration views

Decoration并不是数据驱动的，和UICollectionViewDataSource毫无关联

UICollectionViewDataSource

数据源协议回到了三个问题

  • How many sections does the collection view contain?
  • For a given section, how many items does a section contain?
  • For a given section or item, what views should be used to display the corresponding content?

UICollectionViewDataSource的实现是UICollectionView的核心。无论从设计思维上来看，和实际应用的角度。

UICollectionViewFlowLayout

UITableView中，UITableViewCell都是单一的排布顺序。即从上到下。顺序排布。而UICollectionView中，可以实现非常灵活的排布。依靠的类就是UICollectionViewFlowLayout类。它的任务就是1.配置滚动方向。2.配置cell间隔。3.配置supplementary尺寸。其实使用UICollectionViewFlowLayout方法十分简单。

FlowLayout流程

  • 创建flow layout对象和赋值给collectionview对象（注意手写代码的时候，创建collectionview必须传入这个对象）
  • 配置cell的宽度和高度
  • 创建行间隔和item间隔
  • 如果需要设置headers和footers，指定它们的尺寸。

collectionview基本流程

实现一个collectionview，最少意义上要做这些步骤。

0.创建UICollectionViewFlowLayout对象
1.创建UICollectionview对象        
2.设定UICollectionview对象的数据源委托，如有需要，设定视图委托，和布局委托
3.实现UICollectionview对象的数据委托协议方法。如有需要，设定视图委托，和布局委托方法。

侧边栏的实现

发表于 2016-03-20

本文大部分思路参考小王子

基本框架

类似容器视图控制器的思想。对于一个侧边栏，至少需要三个角色控制器。

  1. 侧边栏视图控制器
    提供侧边栏的显示内容。用户点击侧边栏的按钮可以调用管理控制器隐藏侧边栏，显示主要展示内容。
  2. 主要展示内容控制器
    多个内容控制器。和侧边栏上的按钮对应。
  3. 管理容器控制器
    侧边栏和主要展示内容控制器的视图都添加到管理控制器的视图上。真正决定显示内容给用户看的控制器。

可以看出，三种角色控制器中，管理控制器是开发设计的核心。定义这个类叫做ZZSlideBarViewController.

不太好的思路

可以考虑仅仅使用一个视图控制器。左边放侧边栏视图。右边放主内容视图。这样的的话，这个控制器实际上承担太多的职责。会使用类臃肿庞大，而且很难重用。

设计思路

  1. 先不要求可以点击tab上的内容跳转不同的视图控制器，而是仅仅实现一个内容控制器的版本。而且动画是平行移动， 不是类似QQ侧边栏的缩放效果。
  2. 再使用多个控制器的版本。
  3. 最后实现可平行移动的控制器

简单工厂模式概述

简单工厂模式（简称工厂模式）属于创建型模式，又叫做静态工厂方法，也就是说使用这种模式是为了创建对象。

角色

简单工厂模式中有几种类型的角色:
工厂角色（Factory）:创建具体产品的类，负责创建的逻辑
抽象产品（Product）角色: 为具体产品提供统一接口,可以是协议，也可以是抽象类
具体产品（Concrete Product):角色要创建的对象的对应的类
客户端: 调用工厂角色创建具体产品。
显然，工厂角色的创建是这种模式的核心。下面是示意图

演示代码

地址：https://github.com/zhou18520786640/Factory

Factory 类

#import "ZZProduct.h"

@interface ZZProductFactory : ZZProduct
+ (ZZProduct *)createProductWithclassName:(NSString *)className;
@end

#import "ZZProductFactory.h"

@implementation ZZProductFactory
+ (ZZProduct *)createProductWithclassName:(NSString *)className {
    Class class = NSClassFromString(className);
    ZZProduct *product = [[class alloc] init];
    return product;
}
@end

使用场景

在下面一些场景下可以考虑使用简单工厂模式

  1. 需要创建具体产品的地方很多，如果不使用工厂类创建，每个客户端地方都需要｀if(…)else(…)`来创建.代码不优雅。实际上这部分逻辑可以全部封装到工厂类来实现
  2. 创建每一个具体产品后还需要做一些繁琐的配置工作。在客户端做这些配置工作也会让代码很难看。
  3. 1还有一个问题，就是又新添加一个具体产品的时候。如果客户端有很多地方，需要很多地方都继续添加else(…)创建具体的产品。

补充

在iOS中，可以根据类名（NSString）生成对应的Class对象。然后根据Class对象生成对应的实例对象
应用这种方法，可以避免在工厂类中写if…else.

Class class = NSClassFromString(className);
ZZProduct *product = [[class alloc] init];
