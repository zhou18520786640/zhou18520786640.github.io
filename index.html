UITableViewCell 动画

发表于 2016-06-19

目标与思路

希望做如上图的动画。观察这个动画，每一个cell从左到右进入眼帘。下面的cell比上面的cell到底最终位置的时间更久。现在有两种思路。
1.每个cell动画启动时间不同。动画的执行时间相同
2.每个cell动画启动时间相同。后面的动画执行时间更久。

方案2更简单，因为很难实现每一个动画的启动时间固定时间间隔

执行动画的时机

UITableView的委托方法来执行动画。

- (void)tableView:(UITableView *)tableView willDisplayCell:(UITableViewCell *)cell forRowAtIndexPath:(NSIndexPath *)indexPath

是tableViewCell的动画的最佳时机

该方法中可以使用[UIView animateWithDuration:animations:]。要注意的是，我们的动画只是在第一次cell出现在屏幕上的时候执行。在其他时候。如滑到上面后又滑下来的时候是不该被调用的。我们可以用一个全局变量。保证动画执行次数和屏幕的显示个数一致。

完整demo

//
//  ViewController.m
//  Animation
//
//  Created by zhou on 16/6/15.
//  Copyright © 2016年 jeff.zhou. All rights reserved.
//

#import "ViewController.h"
#import "ZZCell.h"

@interface ViewController () <UITableViewDelegate, UITableViewDataSource>
@property (nonatomic, strong) UITableView *tableView;

@property (nonatomic, assign) NSInteger count;
@end

@implementation ViewController

- (void)viewDidLoad {
    [super viewDidLoad];
    [self setupTableView];
}

- (void)setupTableView {
    CGRect rect = CGRectMake(0, 0, 80, [UIScreen mainScreen].bounds.size.height);
    UITableView *tableView = [[UITableView alloc] initWithFrame:rect];
    tableView.delegate = self;
    tableView.dataSource = self;
    [tableView registerClass:[ZZCell class] forCellReuseIdentifier:@"cell"];
    self.tableView = tableView;
    self.count = 0;
    [self.view addSubview:self.tableView];
}

- (void)tableView:(UITableView *)tableView willDisplayCell:(UITableViewCell *)cell forRowAtIndexPath:(NSIndexPath *)indexPath {

    // 16是一屏显示的cell个数，这个方法是要保证第一次出现cell的时候动画
    if (self.count > 16) {
        return;
    }
    self.count++;
    CGPoint oldPoint = cell.frame.origin;
    CGPoint newPoint = oldPoint;
    // 第一个偏移量是 0.1*[UIScreen mainScreen].bounds.size.width
    // 第二个偏移量是 0.2*[UIScreen mainScreen].bounds.size.width
    // 第i个偏移量是 i * 0.1 * [UIScreen mainScreen].bounds.size.width
    // 保证速度相同 ，每一个的速度都是[UIScreen mainScreen].bounds.size.width /每点
    newPoint.x = - 80 -indexPath.row * 80 * 0.15;
    cell.frame = CGRectMake(newPoint.x, newPoint.y, cell.frame.size.width, cell.frame.size.height);
    [UIView animateWithDuration:(0.5 + 0.125 * indexPath.row) animations:^{
        cell.frame = CGRectMake(oldPoint.x, oldPoint.y, cell.frame.size.width, cell.frame.size.height);
    }];
//    NSLog(@"ld", indexPath.row);
}

- (NSInteger)tableView:(UITableView *)tableView numberOfRowsInSection:(NSInteger)section {
    return 20;
}

- (UITableViewCell *)tableView:(UITableView *)tableView cellForRowAtIndexPath:(NSIndexPath *)indexPath {
    ZZCell *cell = [tableView dequeueReusableCellWithIdentifier:@"cell" forIndexPath:indexPath];
    return cell;
}


@end

Tip:一次git冲突文件的简单解决

发表于 2016-06-14

project.pbxproj的文件冲突如何解决？

今天使用pull代码的时候产生了project.pbxproj的文件冲突。按照惯常的方法打开project.pbxproj手动去搜索<<<<这样类似文本，删掉你不需要的，移动代码到需要的位置。但是实际过程中非常不理想。因为project.pbxproj的可读性比较差。因为记得自己本地的修改方法是创建了几个新的文件和目录。在同事的建议下，标记project.pbxproj为服务器的版本为准。然后这个时候再运行xcode,发现project.pbxproj不再报错。而是提醒找不到一些头文件。因为相关的头文件和实现文件虽然已经在了工程中。却没有被project.pbxproj所管理，这个时候只需要自己在工程所在的目录搜索后自己手动添加到xcode的管理中即可。

Tip:UIButton的imageView的显示

发表于 2016-06-11

场景

当我们在为视图控制器添加leftBarItem或者rightBarItem的时候，视觉工程师给我们的图的位置总不是苹果设计好的位置。而leftBarItem的customview又不是自己能决定的。这时候我们可以利用UIButton的一些属性来达到想要的位置和效果。

setImage:...与setBackgroundImage:...

使用setImage:会保持image大小不变，让内部的ImageView和image一样大，并将ImageView居中显示在button上
使用setBackgroundImage也会让image大小和button一样大，并且让imageview也和image，button一样。
总结来说。一个不会拉伸图像。一个会拉伸图像。但无论拉伸与否，都会让imageView的尺寸和image一样。
如图1，图2

imageEdgeInsets调整位置

UIEdgeInsets 是一个结构体,描述了一个视图与父视图的直接的间距。

typedef struct UIEdgeInsets {
    CGFloat top, left, bottom, right;  // specify amount to inset (positive) for each of the edges. values can be negative to 'outset'
} UIEdgeInsets;

正数表示离父视图内部的距离。负数表示离父视图外部的距离。top,bottom都是垂直维度的变量。left,right是水平维度的变量。我们只需要改变垂直维度的其中一个变量即可影响垂直位置。同理改变left就可以改变水平维度（right = 0 ） 。当我们使用setImage:设置imageView后，发现imageView水平位置不满足要求,可以这样设置

button.imageEdgeInsets = UIEdgeInsetsMake(28,0,0,0);

28将覆盖原来相对button垂直居中的布局。直接将让imageView垂直方向下与button间隔28.因为我们让top值为28.因为不再需要设置bottom.(bottom = 0)依然不变。注意这个时候不会对button进行拉伸。同时也不会改变水平位置。依然水平居中。就如图3。

如果要改变水平方向。可以设置

button.imageEdgeInsets = UIEdgeInsetsMake(0,28,0,0);

完整demo

#import "ViewController.h"

@interface ViewController ()

@end

@implementation ViewController

- (void)viewDidLoad {
    [super viewDidLoad];

    {   // 不自动铺满，image维持原来的位置,并且居中
        UIButton *button = [UIButton buttonWithType:UIButtonTypeCustom];
        button.frame = CGRectMake(20, 20, 44, 44);
        [button setImage:[UIImage imageNamed:@"add"] forState:UIControlStateNormal];
        button.backgroundColor = [UIColor redColor];
        [self.view addSubview:button];
    }

    {   // 自动铺满，不管uiimage 本身的大小
        UIButton *button = [UIButton buttonWithType:UIButtonTypeCustom];
        button.frame = CGRectMake(20, 70, 44, 44);
        [button setBackgroundImage:[UIImage imageNamed:@"add"] forState:UIControlStateNormal];
        button.backgroundColor = [UIColor redColor];
        [self.view addSubview:button];
    }


//    UIKIT_STATIC_INLINE UIEdgeInsets UIEdgeInsetsMake(CGFloat top, CGFloat left, CGFloat bottom, CGFloat right) {

    {   // image维持原来的尺寸.
        UIButton *button = [UIButton buttonWithType:UIButtonTypeCustom];
        button.frame = CGRectMake(20, 130, 44, 44);
        [button setImage:[UIImage imageNamed:@"add"] forState:UIControlStateNormal];
        button.backgroundColor = [UIColor redColor];

        // top 让他不再遵从垂直方向的居中。而是以button顶部内边距28
        // left,right等于0 表示依然遵从水平方向居中
        button.imageEdgeInsets = UIEdgeInsetsMake(28,0,0,0);
        [self.view addSubview:button];
    }


    {   // image维持原来的尺寸.
        UIButton *button = [UIButton buttonWithType:UIButtonTypeCustom];
        button.frame = CGRectMake(20, 200, 44, 44);
        [button setImage:[UIImage imageNamed:@"add"] forState:UIControlStateNormal];
        button.backgroundColor = [UIColor redColor];
        // left 让他不再遵从水平方向的居中。而是以button左边内边距28
        //  注意left有值就不再是居中后再加上left的距离。而是直接以left距离为依据
        //  top,bottom= 0 表示依然遵从垂直方向居中
        //   在image"内部"插入一个你“看不见”的内边距
        button.imageEdgeInsets = UIEdgeInsetsMake(0,28,0,0);
        [self.view addSubview:button];

        UIView *blueView = [[UIView alloc] init];
        blueView.backgroundColor = [UIColor blueColor];
        blueView.frame = CGRectMake( 0, 14, 28, 16);
        [button addSubview:blueView];
    }

}

一周实践小结

发表于 2016-06-04

主动实践的被动学习

在实际工作中会遇到一些知识的盲点和积累一些经验，这里做一点简单的总结。

不要试图使用button.titleLabel.text来读写button的显示内容？

我们都知道使用button要显示一个值的时候。不能类似cell一样调用cell.textLabel.text = @"123123";来调用
button.titleLabel.text = @"rrr"(虽然有时候有效),正确的写的方法应该是 [button setTitle:@"我是按钮" forState:UIControlStateNormal];.

同时读的操作也不该使用button.titleLabel.text来获取其显示内容。[button titleForState:UIControlStateNormal]是更加稳妥的方案。 button.titleLabel.<#property#>是用于设置title的颜色和字体大小等。如button.titleLabel.textColor等。

手写collectionview一定要在初始化对象的时候传入UICollectionViewFlowLayout对象

只有在iOS8下才可以使用自动布局+xib下来自动布局tableViewcell

拖入xib后的控件视图宽度不一定等于屏幕宽度？

拖入xib后的视图控件的宽度不一定等于当前屏幕的宽度。
需求场景如下。我现在有一个ImageView 几个UILabel.放置顺序是从上到下。imageView对齐。其他Label和imageView左对齐。我设定UIImageView居中后。只是把Label放在和uiimageview同样x的位置。然后设定它们与父视图的位置的约束。因为屏幕的宽度和xib view宽度不一致。程序运行后，label还是保持原来的位置。uiimageview为了居中，会向左偏移。实际上uiimageview的位置才是我真正需要的位置。我应该设定uiimageview后。其他label都和uiimageview左对齐才对。并且保险的方案是将xib中的根视图的宽度调节到一致。

为什么RAC中叫做双向绑定而不是双向赋值？

因为rac中有一个响应式概念。对一个对象的属性做持续的观察而并不是简简单单的只是一次赋值操作
如命令式编程中
label.text = model.text; // model的text再赋予新的值不会影响label的显示值。
rac中对model的属性进行监听。然后将信号的输出反馈给label，就是绑定model的属性。因为model的text在model生命周期内任何一次变化都会反馈给label.而双向绑定就是说身为视图角色的view的属性也可以被监听。

xib和自动布局不适用一些场景？

复杂的动画不适用于。cell的高度计算在iOS8以下。UITableViewCell,UICollectionViewCell的变动很剧烈的场景。复杂的布局场景不太适用于。复杂的视图层次也不利于调试。

如果工时的估计还缺乏必要的关键条件。可以给PMO说一下要等会儿再给他答复。

自定义导航条的一些方法？

1.可以隐藏系统的导航条。然后贴一个自己的view,注意要在合适的时机隐藏或者显示导航条。
2.使用运行时。这个专门写一篇文章分析

简单的东西一学就会，复杂的东西一学不会？

简单的东西我一学就会，一些小技巧，小tips用的得心应手。但是这些都是术的范畴。对于复杂的东西，如架构设计 运行时之类的每一次只是浅尝一下就放在一边了。这种可以上升到艺的东西还是按照学习小tip的方法来实践显然是没有效果的的。要花时间来实践提高我自己艺的学习方法。现在暂时想到的方法步骤是1.看国内demo 2.看国内教程 3.实践写玩具应用，同时不断google 4.看官方文档 5.在1，2，3，4过程中不断问老司机。6.在1，2，3，4，5中不断写个人技术文档 7.把任务或者分阶段来设定目标

UITableView复用机制浅析

发表于 2016-06-01

UITableViewCell的复用机制疑惑

无论是刚接触iOS不久的新人，还是拥有几年从业经验的老司机，都写过下面的代码。

// initTableView的时候
[self.tableView registerClass:[UITableViewCell class] forCellReuseIdentifier:@"cell"];
// UITableViewDataSource回调中
static NSString *reuseIdentifier = @"cell";
UITableViewCell *cell = [tableView dequeueReusableCellWithIdentifier:reuseIdentifier forIndexPath:indexPath];

或者在iOS6更早的版本中。使用如下的代码

// UITableViewDataSource回调中
static NSString *reuseIdentifier = @"cell";
UITableViewCell *cell = [tableView dequeueReusableCellWithIdentifier:reuseIdentifier forIndexPath:indexPath];
if (cell == nil) {
        UITableViewCell *tableViewCell = [[UITableViewCell alloc] initWithStyle:UITableViewCellStyleDefault reuseIdentifier:@"cell"];

}

上面两种方法，并明显区别。实现目的一样。只是换了不同的API而已。所谓复用，就体现在UITableViewCell *cell = [tableView dequeueReusableCellWithIdentifier:reuseIdentifier forIndexPath:indexPath];。这里有一些概念。identifier每创建一个cell必须附带一个identifier标识自己。ReusableCell表示这个cell是复用的。那么何为复用。实际上就是反复循环使用呗。既然要反复循环使用。就必须将可复用的对象放在一个数据结构中。需要的时候取出来放在显示区域。所谓显示区域的cell.我们翻看UITableView的头文件，看到一个@property (nonatomic, readonly) NSArray<__kindof UITableViewCell *> *visibleCells.显示区域的cell全部放在这个数组中。有了显示区域的cells.dequeue表示苹果实现了一个队列的结构。既然有出队列。那么必然有入队列。何时入，何时出？

总结来说。我们需要回到下面几个问题。
1.苹果是用什么数据结构来保存复用的cells.
2.何时将cell加入复用的数据结构。何时从这个复用结构中移除？

隐藏的_reuseTableViewCells

实际上,苹果将UITableView中有个私有变量叫做_reuseTableViewCells。利用运行时的api我们可以将其打印出来。

- (void)tryMember
{
    const char *cellName = "_reusableTableCells";
    Ivar var = class_getInstanceVariable([UITableView class], cellName);

    const char *memberName = ivar_getName(var);
    const char *memberType = ivar_getTypeEncoding(var);
    NSLog(@"%s----%s", memberName, memberType);

}

发现_reuseTableViewCells是一个cell为value的可变字典。它是作为reuseCell的队列结构的关键。

cell在_reuseTableViewCells和visibleCells的相互迁移

我有100条数据，屏幕可以显示17个cell.

  1. 在tableView首次初始化并添加到其父视图后。会自动reloadData
    实际上。UITableView只会调用
    - (UITableViewCell *)tableView:(UITableView *)tableView cellForRowAtIndexPath:(NSIndexPath *)indexPath17次。同时每次调用这个方法都会调用UITableViewCell *tableViewCell = [[UITableViewCell alloc] initWithStyle:UITableViewCellStyleDefault reuseIdentifier:@"cell"];这17个cell都是用户可以看到的。其实也都在visibleCells中。而执行[tableView dequeueReusableCellWithIdentifier:reuseIdentifier forIndexPath:indexPath];都会为nil.即这个时候没有cell在_reuseTableViewCells中。所以出队列操作都是nil.

  2. 当向上滚动时候。这个时候会先出_reuseTableViewCells中去取。发现为nil.那么回再调用一次UITableViewCell *tableViewCell = [[UITableViewCell alloc] initWithStyle:UITableViewCellStyleDefaultreuseIdentifier:@”cell”]。并自动将cell加入到visibleCells中。

  3. 继续往上滑动。第一个cell退出了显示区域。这个时候就从visibleCells中移除。同时将其放入_reuseTableViewCells。这个过程对_reuseTableViewCells而言。就是入队列了。
  4. 再往上滑动。新出来的cell其实不再是alloc出来。而是从_reuseTableViewCells中出队列来的。一旦出队列。就显示到屏幕上。即加入到visibleCells中。

总结来说。任何一个时间点。cell要么在visibleCells中。要么在_reuseTableViewCells中。它们之间会因为隐藏和显示，有限个的cell（屏幕个数+1）不断的在这个两个对象中迁移。

cell虽复用，数据常更新

- (UITableViewCell *)tableView:(UITableView *)tableView cellForRowAtIndexPath:(NSIndexPath *)indexPath
cell虽然复用，但是新出来的cell的展示数据并不是以前的数据。是因为上述方法会更新显示出来的（实际是_reuseTableViewCells中的）显示数据。
